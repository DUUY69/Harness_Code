Mọi thứ đang thay đổi rất nhanh. Đừng lo — nếu hiểu bản chất, gọi tên gì cũng được. Trong bài này (và khóa 12 lecture đi kèm), chúng ta cùng trả lời: Harness Engineering là gì, vì sao model không phải thứ quan trọng nhất, và làm sao biến agent thành công cụ kỹ thuật đáng tin.

1/ Harness Engineering là gì?

Harness Engineering là kỹ thuật xây dựng toàn bộ môi trường xung quanh AI model — tools, quyền truy cập, bộ nhớ, feedback loops, guardrails, quản lý context, handoff giữa sessions — tất cả mọi thứ trừ model weights.

“Mỗi khi agent mắc lỗi, bạn không cầu nguyện nó làm tốt hơn lần sau — bạn xây giải pháp để nó không bao giờ mắc lỗi đó nữa.” — Mitchell Hashimoto

90% người dùng AI vẫn chỉ tối ưu prompt và chọn model xịn hơn, trong khi thiếu phần quan trọng nhất: thiết kế môi trường cho agent vận hành.

2/ Tại sao model không phải thứ quan trọng nhất?

Cùng một model, cùng task, cùng compute — chỉ đổi thiết kế môi trường — hiệu suất tăng 64% (paper SWE-agent, Princeton NLP, NeurIPS 2024). Khác biệt giữa công cụ dùng được và vô dụng đến từ environment design, không phải model mới.

Chỉ thay đổi Agent-Computer Interface (harness) — không nâng cấp model weights

“The model is what thinks. The harness is what it thinks about. Getting that distinction right is the entire game.” — Rohit Verma

3/ Bài học từ SWE-agent: thiết kế interface cho agent

AI agent không suy nghĩ như người: token tuần tự, working memory giới hạn, dễ nhiễu bởi context thừa. ACI (Agent-Computer Interface) thiết kế xung quanh hạn chế đó — không chống lại chúng.

  • Search có giới hạn — tối đa ~50 kết quả, ép refine query
  • File viewer có dòng số — ~100 dòng/lần, Goldilocks zone
  • Editor + linter — reject syntax error ngay, chặn failure dây chuyền
  • Context management — nén history cũ, giữ signal

4/ Anthropic: hai failure mode & kiến trúc multi-agent

Pattern 1 — Làm quá nhiều một lúc: ôm cả project, hết context, session sau nhận đống code dở dang.

Pattern 2 — Tuyên bố chiến thắng quá sớm: thấy có code → kết luận “xong” — suy luận hợp lý từ thông tin thiếu.

Giải pháp khóa học map trực tiếp: WIP=1, feature list JSON, init phase tách riêng, evaluator tách khỏi generator — và pass-state gating: agent không tự đánh dấu DONE.

5/ Năm subsystem — khung harness đầy đủ

Harness = Instructions + Tools + Environment + State + Feedback. Thiếu một subsystem → agent luôn “khó dùng”. Feedback (verify commands) thường ROI cao nhất.

6/ Ba lớp termination — chặn “xong” giả

Unit test pass ≠ task complete. Khóa học ép 3-layer validation: lint/type → runtime behavior → system E2E. Kết hợp pass-state gating trên feature_list.json.

8/ Tạm kết

Nếu chỉ nhớ một câu: Model là thứ suy nghĩ. Harness là thứ nó suy nghĩ VỀ. Harness mới quyết định kết quả cuối cùng — kể cả khi benchmark model đã rất cao.

Harness Engineering là systems thinking cho môi trường agent: API design, error handling, testing strategy — chỉ khác domain: thiết kế cho LLM thay vì cho người.
