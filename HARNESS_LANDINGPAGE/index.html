Nỗi đau khi dùng agent không harness

Agent không yếu vì model — mà vì môi trường không ràng buộc, không verify và không giữ được state giữa các session.

50–60%

Benchmark ≠ thực tế

SWE-bench Verified chỉ ~50–60% pass rate dù điều kiện lý tưởng. Yêu cầu mơ hồ, không test sẵn — tỷ lệ thành công còn thấp hơn nữa.

Context

Mất thread giữa session

Session mới không nhớ quyết định, file đã sửa, test đang pass hay fail. Mỗi lần mở lại tốn 15+ phút chỉ để “đoán lại” trạng thái.

Sớm

Tuyên bố “xong” quá sớm

Unit test pass, code nhìn ổn — nhưng E2E chưa chạy, migration lỗi, config thiếu. Agent overconfident; harness phải thay “cảm giác” bằng verify.

Scope

Overreach & under-finish

Một prompt rộng → 12 file, 800 dòng, không feature nào pass E2E. Attention bị chia; WIP>1 khiến không task nào hoàn thành đúng nghĩa.

Hai lớp giải pháp

Harness Engineering giải quyết cả “chạy đúng ngay lúc này” và “giữ độ tin cậy qua nhiều session dài”.

Control layer

Ràng buộc hành vi: WIP=1, feature list, completion evidence, 3-layer termination check.

Agent không tự giới hạn — harness phải vẽ ranh giới task và định nghĩa “done” bằng lệnh chạy được. Quan trọng hơn: pass-state gating — agent không có quyền tự đánh dấu DONE; chỉ harness (hệ thống chạy test) mới được chuyển feature sang trạng thái passing.

  • Feature list: behavior + verification command + state
  • Pass-state gating — agent submit verify, harness quyết định done
  • 3-layer termination + E2E — không shortcut unit test
Continuity layer

Repo là system of record: state persistence, init phase, observability và clean handoff.

Kiến thức không có trong repo = không tồn tại với agent. PROGRESS.md, DECISIONS.md, sprint contract và session exit checklist giữ continuity qua session dài.

  • AGENTS.md ngắn — router, không phải bách khoa
  • Initialization tách khỏi implementation
  • Clean state 5 điều kiện trước khi kết thúc session

Vòng lặp harness

Từ mục tiêu rõ ràng đến handoff sạch — 4 bước lặp lại mỗi session.

01

Clear objective

Định nghĩa task trong AGENTS.md, đọc feature list, chạy init.sh — session bắt đầu từ trạng thái đã biết, không đoán mò.

02

Run task

Agent thực thi một feature active (WIP=1). Code, test, commit theo từng đơn vị nguyên tử — không ôm nhiều việc cùng lúc.

03

Verify & QA

Harness chạy lint, test, E2E — không tin lời agent. Pass-state gating: chỉ khi verification command pass thì harness mới unlock trạng thái passing; agent không tự tick done.

04

Handoff sạch

Cập nhật PROGRESS.md, dọn artifact tạm, đảm bảo build + test pass. Session sau đọc repo là tiếp tục được — không mất 30–50% thời gian chẩn đoán lại.

5 subsystem harness

Harness = mọi thứ ngoài model weights. Thiếu một subsystem là trải nghiệm agent sẽ luôn “khó dùng”.

01 Instructions

AGENTS.md / CLAUDE.md — entry router

Overview, lệnh chạy, hard constraints, link tới topic docs. ~50–200 dòng.

  • Repo IS the spec
  • Tách file — tránh lost in the middle
  • Fresh session test
02 Tools

Shell, file, test — least privilege

Agent cần hành động được; không có tools thì chỉ là chatbot.

  • Cursor, Claude Code, Codex patterns
  • Không tắt shell hoàn toàn
  • Tool output là input quan trọng
03 Environment

Reproducible runtime

pyproject.toml, devcontainer, Makefile — môi trường tự mô tả.

  • make setup / make test / make check
  • Init phase riêng — không trộn implement
  • Template > empty directory
04 State

Continuity qua session

PROGRESS.md, DECISIONS.md, git commits như checkpoint miễn phí.

  • Rebuild cost: 15 phút → ~3 phút
  • Context anxiety & compaction
  • ACID cho agent state
05 Feedback — ROI cao nhất

Verification commands trong AGENTS.md: pytest, mypy, ruff, E2E — feedback subsystem thường đầu tư ít, hiệu quả lớn nhất.

Observability (runtime + process), sprint contract, evaluator rubric, review feedback promotion — harness tự mạnh dần theo từng session.

  • 3-layer termination: lint → behavior → system E2E
  • Error message viết cho agent — có bước sửa cụ thể
  • Clean state 5 điều kiện khi exit session

Câu hỏi thường gặp

Harness Engineering khác gì “prompt engineering”?

Đây có phải khóa bán gói SaaS không?

Không. Đây là khóa học kỹ thuật: lecture, project và template. Phần “Chương trình học” mô tả lộ trình nội dung — không phải bảng giá subscription.

AGENTS.md có thay được model đắt hơn không?

Nhiều case: có. Cùng model, thêm AGENTS.md + verification commands có thể tăng success rate từ ~20% lên 80%+. Đổi model là option đắt nhất — check harness trước.

Pass-state gating nghĩa là gì?

Agent không được tự đổi state feature sang “passing”. Agent chỉ submit verification request; harness chạy lệnh test và mới quyết định done. Đây là cơ chế chặn false victory — không tin lời “đã xong” của model.

WIP=1 có quá chậm không?

Ngược lại: ít code hơn nhưng nhiều feature pass E2E hơn. Case 8 features: WIP=1 đạt 87.5% completion vs 37.5% khi agent ôm 5 feature cùng lúc.

Bắt đầu từ đâu nếu mới học?

Lecture 01 → Lecture 02 → Project 01. Song song: đặt AGENTS.md vào repo root, thêm make test / make check, rồi chạy fresh session test.
