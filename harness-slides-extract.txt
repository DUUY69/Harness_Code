LECTURE 10 | Only a Full Pipeline Run Counts as Real Verification
--- Slide 2 ---
THE PROBLEM | When "Done" Isn't Done | A Familiar Scenario | You ask the agent to add a file export feature to an Electron app. It writes the renderer component, the preload script, and the service layer logic. Unit tests for every component pass. The agent says | "done." | You actually click the export button — the file path format is wrong, the progress bar doesn't respond, and exporting large files leaks memory. | The Core Problem | Each part looks "correct" on its own, but problems surface the moment they are wired together. Google's Testing Pyramid tells us that a large base of unit tests is essential, but stopping there means you will | systematically miss component interaction issues. | For AI coding agents this problem is even worse, because agents tend to run only the fastest tests and then declare completion. | Five Defects, Zero Caught | In the Electron file export case, unit tests passed on every component — yet five component boundary defects were present, and unit tests didn't catch a single one. | Only end-to-end testing can prove the absence of system-level defects.
--- Slide 3 ---
BLIND SPOTS | The Blind Spots of Unit Tests | The design philosophy of unit testing is | isolation | : mock dependencies, focus on the unit under test. This philosophy makes unit tests fast and precise — but it also creates systematic blind spots. | Interface Mismatch | The renderer passes a relative file path, but the preload script expects an absolute path. Both unit tests use mocks and pass. The problem only surfaces end-to-end. | State Propagation Errors | A database migration changes the schema, but the ORM caching layer holds old entries. Unit tests spin up a fresh mock each time and never expose this. | Resource Lifecycle Issues | File handles, database connections, and network sockets span multiple components. Unit tests never surface resource contention or leaks. | Environment Dependency | Code behaves correctly in the mocked test environment but fails in the real environment due to configuration differences, latency, or service unavailability.
--- Slide 4 ---
BEHAVIOR SHIFT | End-to-End Testing Not Only Changes Results — It Changes Behavior | This is something many people overlook: when an agent knows its work will be validated by end-to-end tests, its | coding behavior shifts. | Considering Component Interactions | While writing code the agent starts asking "how does this interface connect with the upstream?" rather than focusing on a single function in isolation. | Respecting Architectural Boundaries | In systems with architectural constraints, end-to-end tests force the agent to follow boundary rules that would otherwise be silently ignored. | Handling Error Paths | End-to-end tests typically include failure scenarios, which pushes the agent to think about exception handling rather than only the happy path. | End-to-end testing is not just a quality gate — it is a | forcing function | that shapes how agents reason about the code they produce.
--- Slide 5 ---
TESTING PYRAMID | Testing Pyramid and Review Feedback Promotion | The Adequacy Gradient | Defects detectable by unit tests | ≤ | integration tests | ≤ | end-to-end tests. Detection capability increases with each layer. | Agent-Oriented Error Messages | OpenAI emphasizes in its Codex engineering practices: | error messages written for agents must include fix instructions. | Instead of: | "Direct filesystem access in renderer" | Write: | "Direct filesystem access in renderer. All file operations must go through the preload bridge. Move this call to preload/file-ops.ts and invoke it via window.api." | This turns architectural rules into an | auto-correction loop | — error messages don't just tell you what went wrong, they tell you how to fix it.
--- Slide 6 ---
CORE CONCEPTS | Core Concepts | Component Boundary Defects | Component A and B each pass their unit tests, but their interaction produces incorrect behavior. This is the category of issue that end-to-end testing excels at catching. | Testing Adequacy Gradient | Defects detectable by unit tests ≤ integration tests ≤ end-to-end tests. Detection capability increases with each layer of the testing hierarchy. | Architectural Boundary Enforcement | Turning rules from architecture documents into executable, automated checks — going from "written on paper" to "running in CI" on every commit. | Review Feedback Promotion | Converting recurring code review comments into automated tests. Every time a new category of repeated issue is found, add a rule — the harness grows stronger automatically.
--- Slide 7 ---
HOW TO DO IT | How to Implement End-to-End Validation | Agent Error Messages | Executable Checks | Add E2E Layer | Boundaries | A structured four-step approach to making full-pipeline verification a mandatory part of every agent workflow. | 01 | Define Architectural Boundaries First | The prerequisite for end-to-end testing is that the system has clear boundaries. OpenAI adopted a "Layered Domain Architecture" — Types → Config → Repo → Service → Runtime → UI. | Enforce invariants; don't micromanage implementation. | 02 | The Harness Must Include an E2E Layer | Make it explicit: for tasks involving cross-component changes, passing end-to-end tests is a prerequisite for completion. Skipping any required level = Not Complete. | 03 | Turn Architectural Rules into Executable Checks | Every architectural constraint should have a corresponding test or lint rule. For example: | grep -r "require('fs')" src/renderer/ && exit 1 || echo "OK" | 04 | Establish a Review Feedback Promotion Process | Every time you discover a new category of agent error during code review, turn it into an automated check. A month later your harness will be far stronger than it was at the start.
--- Slide 8 ---
REAL-WORLD CASE | Real-World Case: Electron File Export | Task: | Implement a file export feature in an Electron app — involving renderer process UI, preload script filesystem proxy, and service layer data transformation. Agent declares completion after all unit tests pass. | Defect | Description | Unit Test | E2E Test | Interface Mismatch | Inconsistent file path format between renderer and preload | Missed | Caught | State Propagation | Export progress not sent back to UI via IPC | Missed | Caught | Resource Leak | Large file export handles not released | Missed | Caught | Permission Issue | Different permissions in packaged environment | Missed | Caught | Error Propagation | Service layer exceptions didn't reach UI layer | Missed | Caught | 5 | Defects Found | All 5 were caught by end-to-end tests; unit tests caught none. | 0 | Caught by Unit Tests | Unit tests passed on every component — and missed every boundary defect. | 2s→15s | Test Time Trade-off | From 2 seconds to 15 seconds — perfectly acceptable in an agent workflow.
--- Slide 9 ---
KEY TAKEAWAYS | Key Takeaways | Five principles that define effective end-to-end validation in agent-driven codebases. | Unit Tests Are Systematically Blind to Boundary Defects | Their isolation design is precisely what prevents them from detecting interaction issues. This is a feature, not a bug — but it means unit tests alone are never sufficient. | E2E Testing Changes How Agents Write Code | It shifts agent focus toward integration and boundaries — not just local correctness. | Architectural Rules Must Be Executable | Not written in a document waiting for someone to read them, but automatically checked on every commit via lint rules and automated tests. | Error Messages Must Be Designed for Agents | Include concrete "how to fix" steps to form a self-correcting feedback loop. Turn test failures into autonomous corrections. | Review Feedback Promotion Makes the Harness Stronger | Every captured defect category becomes a permanent line of defense. The harness automatically improves with every session.

Further Reading & Exercises | Further Reading | How Google Tests Software | Whittaker et al. — The classic source of the Testing Pyramid model. | Harness Engineering — OpenAI | Engineering practices for automated enforcement of architectural constraints in agent-first codebases. | Chaos Engineering — Netflix | Basiri et al. — Proactively injecting failures to verify system resilience. | QuickCheck — Claessen & Hughes | Property testing methodology, sitting between example-based testing and formal verification. | Exercises | Cross-Component Defect Detection: | Pick a modification task involving at least three components. Lecture 11 | Making the Agent's Runtime Observable | Agent Harness Series · Course 06
--- Slide 2 ---
THE PROBLEM | Why Observability Matters | The Common Scenario | You ask an agent to implement a feature. It runs for 20 minutes, touches a pile of files, then tells you | "done, but two tests are failing." | You ask why — | "not sure, might be a timing issue." | You ask which critical paths it changed — | "let me look at the code…" | The root cause isn't the agent's capability — it's the harness's | lack of observability | . When an agent executes a task without visibility into actual runtime state, every decision it makes is essentially a guess. | Without observability, agents make decisions under uncertainty, evaluations become subjective judgments, and retries become blind wandering. | The Root Cause | Both OpenAI and Anthropic frame reliability as an | evidence problem | : the harness must expose runtime behavior and evaluation signals in a form that can actually guide the next decision.
--- Slide 3 ---
The Real Cost of Missing Observability | Correct vs. Looks Correct | Code looks right in review — correct syntax, sound logic. But at runtime, boundary-condition errors surface only through traces. Code review shows "what was written"; runtime tracing shows "what actually ran." | Evaluation Becomes Mysticism | Without scoring rubrics and acceptance criteria, evaluators rely on implicit assumptions. The same output receives widely different assessments. Quality evaluation becomes non-reproducible. | Retries Become Blind Guesses | When the agent doesn't know why something failed, its retry direction is random — fixing unrelated code paths while ignoring the real root cause. Every blind retry burns tokens and time. | Session Handoff Information Cliff | Missing observability forces the new session to diagnose system state from scratch. Anthropic's data shows this redundant diagnosis eats up | 30–50% of total session time | .
--- Slide 4 ---
CASE STUDY | A Real Claude Code Scenario | A "planner-generator-evaluator" three-role workflow executing the task | "add dark mode to the app" | — with and without observability. | ❌ | Without Observability | The planner outputs a vague description. The generator implements based on vagueness. The evaluator rejects on implicit standards — "it doesn't feel right." The generator retries blindly. The cycle repeats | 3–4 times | , taking | ~45 minutes | , barely producing an acceptable output. | ✅ | With Full Observability | The planner outputs a sprint contract: which components to modify, verification standards for each, and exclusions. The evaluator cites specific evidence: | "Button color contrast is insufficient (WCAG AA 4.5:1, measured 2.1:1)." | One iteration, | ~15 minutes | , high-quality result. | 3x | Efficiency Difference | The only variable between the two runs was observability. | 45min | Without Observability | 3–4 blind retry cycles to reach an acceptable result. | 15min | With Observability | One well-guided iteration with evidence-backed feedback.
--- Slide 5 ---
Layered Observability | Observability isn't just "add more logging." It operates on | two layers | , and both are essential. | Report Failures | Verify Items | Collect Signals | Implement | Define Task | Runtime signals explain | behavior | ; process artifacts explain | intent | . Neither layer alone is sufficient — together they give the harness the full picture needed to guide reliable agent decisions. | Runtime Observability | System-level signals — logs, traces, process events, health checks. Answers | "what did the system do." | Process Observability | Visibility into harness decision artifacts — plans, scoring rubrics, acceptance criteria. Answers | "why should this change be accepted."
--- Slide 6 ---
CORE CONCEPTS | Core Concepts | Task Trace | A complete decision-path record from task start to completion, analogous to request tracing in distributed systems. Every step the agent takes, along with its context, is recorded — so when something goes wrong, you can replay the entire process. | Sprint Contract | A short-term agreement negotiated | before | coding begins, specifying task scope, verification standards, and exclusions. The core tool for process observability. | Evaluator Rubric | Transforms quality evaluation from subjective judgment into evidence-based structured scoring, enabling different evaluators to reach similar conclusions for the same output. | Layered Observability | System-layer and process-layer observability designed simultaneously and reinforcing each other. Runtime signals explain behavior; process artifacts explain intent.
--- Slide 7 ---
LIMITATIONS | Why Agents Can't Solve This Themselves | You might be thinking: | "Can't the agent just print its own logs?" | The problem runs deeper than that. | 1 | Agents don't know what they don't know | They won't proactively record signals they don't realize they need. Without harness-level constraints, agents only log what they think is important — and what they think is important is usually not enough. | 2 | Log formats are inconsistent | Different sessions use different log formats, making systematic analysis impossible across runs and sessions. | 3 | Process observability can't be solved by logging | Sprint contracts and scoring rubrics are structured artifacts that require harness-level support — adding a few print statements won't cut it. | Observability is a | harness architecture property | , not a feature the agent can retrofit onto itself. It must be designed into the harness from the start.
--- Slide 8 ---
IMPLEMENTATION | How to Build Observability | Build Runtime Signal Collection into the Harness | Don't rely on the agent to print its own logs. Automatically collect: application lifecycle states, feature path execution records, data flow between components, resource utilization anomalies, and full error context. | Implement Sprint Contracts | Before each task begins, the generator and evaluator negotiate a contract defining what to build and what "done" looks like — covering scope, verification standards, and explicit exclusions (e.g., "not handling print styles"). Sprint Contract: Dark Mode Support | Scope | Modify the theme toggle component | Update global CSS variables | Add dark mode tests | Verification Standards | Visual regression tests pass for each component | Main flow end-to-end tests pass | No flash of unstyled content (FOUC) | Exclusions | Not handling print styles | Not handling third-party component dark mode | Establish an Evaluator Rubric | Turn "is it good or not" into quantifiable scoring  Scoring Rubric | Dimension | A | B | C | D | Code correctness | All tests pass | Main flow passes | Partial pass | Build fails | Architecture compliance | Fully compliant | Minor deviations | Obvious deviations | Serious violations | Test coverage | Main + edge cases | Main flow only | Only skeleton | No tests | Standardize with OpenTelemetry | Create a trace for each harness session, a span for each task, and sub-spans for each verification step. This way observability data integrates with standard toolchains like Jaeger and Zipkin.
--- Slide 9 ---
ANTHROPIC RESEARCH | Anthropic's Three-Agent Architecture Experiment | In March 2026, Anthropic published a systematic harness experiment running the task | "build a browser-based DAW using the Web Audio API" | with a three-agent architecture. Phase-by-phase data was recorded: | Agent & Phase | Duration | Cost | Planner | 4.7 min | $0.46 | Build round 1 | 2 hr 7 min | $71.08 | QA round 1 | 8.8 min | $3.24 | Build round 2 | 1 hr 2 min | $36.89 | QA round 2 | 6.8 min | $3.09 | Build round 3 | 10.9 min | $5.88 | QA round 3 | 9.6 min | $4.06 | Total | 3 hr 50 min | $124.70 | Planner | Expands a 1–4 sentence requirement into a full product spec. Instructed to "be bold in scope" and focus on high-level design — not granular implementation, which cascades errors downstream. | Generator | Implements feature by feature. Before each sprint, negotiates a contract with the evaluator defining "done," then self-evaluates and hands off to QA. | Evaluator | Uses Playwright MCP to interact with the running app like a real user. Scores each sprint across four dimensions — product depth, functionality, visual design, and code quality — each with a hard threshold. | Example QA feedback: | "Clips can't be dragged/moved, there's no instrument UI panel (synth knobs, drum pads), and no visual effects editor." | Specific, evidence-backed feedback — not "it doesn't feel right."
--- Slide 10 ---
TAKEAWAYS & FURTHER READING | Key Takeaways | Observability is a harness architecture property | It's not a feature you add after the fact — it must be designed in from the start. | Both observability layers are essential | Runtime signals explain "what happened"; process artifacts explain "why it was done this way." | Sprint contracts front-load alignment | They prevent the generator from building something the evaluator immediately rejects for foreseeable reasons. | Scoring rubrics make evaluation reproducible | Different evaluators produce similar scores for the same output — quality becomes measurable, not mystical. | Missing observability wastes 30–50% of session time | Redundant diagnosis at session handoff is the single largest avoidable cost in long-running agent tasks. | Further Reading | Observability Engineering | — Charity Majors: theory and practice framework for modern observability engineering | Dapper | — Google (Sigelman et al.): groundbreaking practice in large-scale distributed tracing | Harness Design | — Anthropic: introducing sprint contracts and evaluator rubrics | Site Reliability Engineering | — Google: systematic application of observability in production systems

--- Slide 1 ---
Lecture 12 | Leave a Clean Handoff at the End of Every Session
--- Slide 2 ---
OVERVIEW | The Problem: Every Session Leaves a Mess | What Goes Wrong | Your agent runs all afternoon, modifies 20 files, commits the code, and the session ends. The next agent session starts up and immediately discovers: build is broken, tests are red, temporary debug files are scattered everywhere, the feature list hasn't been updated, and progress is completely opaque. | The first 30 minutes of the new session are spent entirely on | "figuring out what the last session actually did." | Why It Matters | Both OpenAI and Anthropic state clearly: | long-term reliability depends on operational discipline, not just single-run success. | The quality of state at the end of each session directly determines the next session's efficiency. | Key Insight | Session exit quality is not optional housekeeping — it is the foundational discipline that separates agents that scale from agents that slowly break down. Every minute saved in the next session was earned by discipline in the previous one.
--- Slide 3 ---
ENTROPY | Entropy Growth Is the Default State | Lehman's laws of software evolution tell us that systems undergoing continuous change will inevitably grow more complex unless actively managed. This is especially true for AI coding agents. Every session introduces changes, and without cleanup at exit, technical debt accumulates exponentially. | The Coffee Cup Effect | During five months of Codex experiments, OpenAI observed: | agents copy patterns already present in the repository, even when those patterns are inconsistent or suboptimal. | The first person leaves a coffee cup in the common area; the second figures "it's already messy" and leaves another; a week later the table is buried under cups. A codebase works the same way. | The OpenAI Solution | The OpenAI team initially spent 20% of every Friday manually cleaning up "AI slop," but this approach clearly doesn't scale. They eventually arrived at a systematic three-part solution: encode golden rules into the repository, establish periodic cleanup workflows, and capture human taste once — then enforce it continuously. | Technical Debt Is a Loan | Technical debt is a high-interest loan. Continuously paying it off in small increments is almost always better than letting it accumulate into one massive payoff event. Background Codex tasks regularly scan for deviations, update quality scores, and open targeted refactoring PRs — most reviewed and auto-merged within a minute.
--- Slide 4 ---
CLEAN STATE | Clean State: More Than "The Code Compiles" | Clean state isn't simply "the code compiles." Building without errors is the most basic requirement — but it is only one of five non-negotiable conditions. | Build Passes | The next session shouldn't have to fix build errors first. Building without errors is the most basic requirement — and not meeting it means the session isn't done. | All Tests Pass | All tests must pass, including tests that existed before the session. The session is responsible for not breaking existing functionality — and this must be verified in CI, not just "works on my machine." | Progress Recorded | Current progress must be recorded in machine-readable artifacts: completed subtasks with passing criteria, in-progress subtasks with current state, and not-yet-started subtasks. Good progress records can reduce session startup diagnostic time by | 60–80%. | No Stale Artifacts | Temporary artifacts — debug logs, temporary files, commented-out code, TODO markers — must be cleaned up, because they increase cognitive load for the next session. | Startup Path Available | The standard startup path must remain functional. Can the next session start working without manual intervention? Environment initialization, codebase loading, context acquisition, task selection — none of these paths can be broken.
--- Slide 5 ---
CORE CONCEPTS | Core Concepts | Clean State | The system must satisfy five conditions at session end — build passes, tests pass, progress recorded, no stale artifacts, startup path available. Missing any one means the session isn't "done." | Session Integrity | Analogous to database transactions — either fully commit and leave a clean state, or roll back to the last consistent state. No middle ground. | Quality Document | An active artifact that continuously records quality ratings for each module. Not a one-time assessment, but a tracker showing whether the codebase is getting stronger or weaker over time. | Cleanup Loop | A regular maintenance session aimed at systematically reducing entropy in the codebase. Not an emergency fix, but routine operations. | Harness Simplification | As model capabilities improve, periodically remove harness components that are no longer necessary. A constraint essential today may be unnecessary overhead in three months. | Idempotent Cleanup | Cleanup operations produce the same result regardless of how many times they run, ensuring cleanup remains safe even in failure-retry scenarios.
--- Slide 6 ---
THE NUMBERS | "Clean Up Later" Means Never Clean Up | The most common mental trap is "no time to clean up this session, I'll do it next time." But the next agent session doesn't know what you left behind — it sees a mess of code and uncertain state. Every session has its own task objectives, so it'll ignore the chaos and start new work on top of it, introducing even more chaos. This is entropy's positive feedback loop. | A project developed with agents for 12 weeks, without a cleanup strategy vs. with one: | After 12 weeks: build pass rate differs by | 29 percentage points | , test pass rate differs by | 34 points | , and new session startup time differs by | 85% | . This is not theoretical — it's an observed difference.
--- Slide 7 ---
HOW TO DO IT | How to Do It: Step 1 — Clean State as Completion Criterion | Define explicitly in the harness: | session completion | = task passes verification AND clean state check passes. Missing either one means the session isn't complete. Write in | CLAUDE.md | : | ## Session Exit Checklist | - [ ] Build passes (npm run build) | - [ ] All tests pass (npm test) | - [ ] Feature list updated | - [ ] No debug code remaining (console.log, debugger, TODO) | - [ ] Standard startup path available (npm run dev) | Missing any single item on this checklist means the session is not complete — no exceptions. "Mostly done" is not done.
--- Slide 8 ---
HOW TO DO IT | How to Do It: Steps 2 & 3 — Dual-Mode Cleanup and Quality Documents | Dual-Mode Cleanup Strategy | Combine two cleanup modes: | Immediate Cleanup (Every Session) | Clean up temporary artifacts created during the session, update feature list state, ensure build and tests pass. This is "reference counting" cleanup — clean up as soon as you're done using something. | Periodic Cleanup (Weekly) | Full-system scan — handle accumulated structural issues, update quality documents, run benchmark tests to detect drift. This is "tracing" cleanup — a comprehensive maintenance pass done on a regular cadence. | Maintain a Quality Document | A quality document is an active artifact that continuously scores each module. New sessions read this document and immediately know where to prioritize — fix the lowest-scoring module first. | ## User Auth Module (Quality: A) | - Verification passing: Yes | - Agent understandable: Yes | - Test stability: Stable | ## Payment Module (Quality: C) | - Verification passing: Partial | - Agent understandable: Difficult | - Test stability: Unstable (2 flaky tests)
--- Slide 9 ---
HOW TO DO IT | How to Do It: Steps 4–6 — Simplify, Idempotency, and Merge Philosophy | Periodically Simplify the Harness | Every harness component exists because the model couldn't reliably do something on its own. But as models improve, these assumptions become outdated. Anthropic's experiments: their initial harness included a sprint-splitting mechanism for Sonnet 4.5. When Opus 4.6 shipped, the mechanism became unnecessary overhead — after removing it, the builder agent worked continuously for over two hours without drifting. | Recommended practice: | Every month, pick one harness component, temporarily disable it, and run benchmark tasks. If results don't degrade, remove it permanently. | Cleanup Operations Must Be Idempotent | Cleanup scripts should be safe to run repeatedly — running them one more time shouldn't produce side effects. | rm -f /tmp/debug-*.log    # -f: no error if absent | git checkout -- .env.local # Restore known state | npm run test               # Verify nothing broke | High Throughput Changes the Merge Philosophy | When an agent opens 3.5 PRs per day, minimizing blocking merge gates is the right call. PRs should be short-lived; test flakiness is usually resolved with subsequent runs. When the cost of fixing is low and the cost of waiting is high, moving fast with fast fixes beats slow confirmation. | Caveat: | This is irresponsible in a low-throughput environment — only applies when agent output far exceeds human attention capacity.

Missing any one element makes the item incomplete and the entire harness loses its shared consensus. | Pass-State Gating | The only way a feature moves from | active | to | passing | is by the verification command executing successfully. This transition is | irreversible | — once | passing | , it cannot go back. | Single Source of Truth + Back-Pressure | All "what needs to be done" information must derive from one feature list — no contradictions with conversation history. The number of features not yet passing is the | back-pressure | the harness exerts on the agent. Zero pressure = project complete.

Why Feature Lists Must Be "Primitives" | Documents vs. Primitives | Documents are for humans to read; primitives are for systems to execute. Documents can be ignored; primitives can't be bypassed. Think of database trigger constraints versus application-layer checks — the former is enforced by the engine, the latter can be accidentally skipped. | Four Harness Components That Depend on It | Scheduler: | Reads states, picks the next | not_started | feature. | Verifier: | Executes verification commands, controls state transitions. | Handoff Reporter: | Auto-generates session summaries from the list. | Progress Tracker: | Tallies state distribution, provides project health metrics. | 5/5 | Enforcement Level | Feature lists as harness primitives play the same role as database-level constraints — the agent cannot bypass them, just as no SQL can skip a trigger constraint.

Every entry must have the triple: | id | , | behavior | (e.g., | POST /cart/items with {product_id, quantity} returns 201 | ), | verification | (the exact runnable command), | state | , and | evidence | (e.g., commit hash + test log). | 02 | Let the Harness Control State Transitions | The agent | cannot | directly change a feature's state to | passing | . It can only submit a verification request. The harness executes the command and decides whether to allow the transition. This is "pass-state gating." | 03 | Write the Rules in CLAUDE.md | Declare: the feature list file path ( | /docs/features.md | ), that only one feature is active at a time, that verification must pass before marking passing, and that the agent must not modify feature list states itself.

--- Slide 2 ---
INTRODUCTION | The Problem: False Victory | A Familiar Scenario | You ask an agent to implement a "password reset" feature. It modifies the database schema, writes the API endpoint, adds the email template, runs the unit tests (all pass), and then confidently tells you | "It's done." | But when you actually try to run it: | The password reset link can't be sent — email service config is missing | The database migration fails halfway, leaving the schema inconsistent | The end-to-end flow has never been executed even once | Why Does This Happen? | The classic 2017 ICML paper by Guo et al. proved: | modern neural networks are systematically overconfident | —the confidence reported by models is significantly higher than their actual accuracy. AI coding agents are no different. | The Core Insight | Agents "feel" done, but in reality they're far from it. Your harness must replace the agent's "feelings" with externalized, execution-based verification.
--- Slide 3 ---
The Slippery Slope | Step 1: Code Looks Okay | Syntax is correct, logic seems reasonable, static analysis shows no obvious errors. | Step 2: Execution Skipped | The harness doesn't enforce comprehensive execution verification, so the agent skips running it or only runs partial tests. | Step 3: Incomplete Testing | Unit tests run but integration tests are skipped; tests run but coverage isn't checked. | Step 4: False Completion | "The code looks fine" is taken as evidence that "the feature is complete." | Information is lost at every step. From task specification to code implementation to runtime behavior, every transformation can introduce bias, and | every skipped verification compounds the information asymmetry.
--- Slide 4 ---
CORE FRAMEWORK | Three-Layer Termination Check | Lint / Typecheck | Static analysis and types pass | Complete User Flow | End-to-end scenario passes | Agent says done | Signal completion request | Tests & Startup | Unit tests and boot checks | The flowchart illustrates two parallel paths: the | correct path | (top) where all three layers must pass before declaring done, and the | failure path | (bottom) where code is written and unit tests are green, but the app never really started, full flow never ran, and config/DB/external service issues all remain hidden — so the agent declares victory too early.
--- Slide 5 ---
DEFINITIONS | Core Concepts | Premature Completion Declaration | The agent asserts the task is complete, but unmet correctness specifications remain. The core problem: the agent judges based on | local, code-level confidence | , while system-level correctness requires | global verification. | Confidence Calibration Bias | A systematic gap between the agent's self-reported completion confidence and actual completion quality. For complex multi-file tasks, this bias is significantly positive—the agent is consistently more confident than its actual performance warrants. | Termination Criteria | A clear, executable set of judgment conditions defined in the harness. The agent must satisfy all conditions before declaring completion. | "Done" shifts from a subjective judgment to an objective determination. | Verification-Validation Dual Gate | The first layer (verification) checks whether the code correctly implements the specified behavior; the second layer (validation) checks whether system-level behavior meets end-to-end requirements. Both must pass before the task is considered complete.
--- Slide 6 ---
Passing Unit Tests ≠ Task Complete | Interface Mismatch | The renderer passes a relative file path to the preload script, but the preload script expects an absolute path. Their respective unit tests both use mocks and both pass. The issue only surfaces during end-to-end testing. | State Propagation Errors | A database migration changes the table schema, but the ORM's caching layer still holds cache entries for the old schema. Unit tests run in a fresh mock environment every time, so cross-layer state inconsistency never surfaces. | Environment Dependency | Code behaves correctly in the test environment (where everything is mocked) but fails in the real environment due to configuration differences, network latency, or service unavailability. | "Refactoring While We're at It" is Poison — Claude Code commonly starts refactoring and optimizing before core functionality has passed verification. Refactoring shifts the boundary between verified and unverified code, potentially breaking previously correct code paths.
--- Slide 7 ---
RESEARCH FINDING | Systematic Bias in Self-Evaluation | Anthropic's 2026 research revealed: | when an agent is asked to evaluate its own work, it systematically provides overly positive assessments—even when a human observer would judge the quality as clearly substandard. | The solution is to separate the "worker" from the "checker." | Architecture | Runtime | Cost | Core Features Working? | Single agent (bare run) | 20 mins | $9 | No (game entities unresponsive to input) | Three agents (planner + generator + evaluator) | 6 hours | $200 | Yes (game is fully playable) | This is the exact same model (Opus 4.5) with the exact same prompt ("build a 2D retro game editor"). The only difference is the harness: from "running bare" to "planner expands requirements → generator implements feature by feature → evaluator performs actual click testing using Playwright." | Source: Anthropic — Harness design for long-running application development
--- Slide 8 ---
How to Prevent Premature Completion | 1. Externalize Termination Judgment | The completion judgment should not be made by the agent itself. The harness independently executes termination validation, using runtime signals as input rather than the agent's confidence. In | CLAUDE.md | : Feature complete = end-to-end verification passed, not "code is written." | 2. Build a Three-Layer Termination Validation | Layer 1: | Syntax and Static Analysis — lowest cost, must pass first. | Layer 2: | Runtime Behavior Verification — tests, startup checks, critical path validation. | Layer 3: | System-Level Confirmation — end-to-end testing, user scenario simulation. | 3. Provide Actionable Error Feedback | Error messages written for agents should include repair instructions. Don't use "Test failed" — use "Test failed: POST /api/reset-password returned 500.
